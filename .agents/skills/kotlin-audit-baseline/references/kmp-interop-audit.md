# KMP structure and native-interop audit rules

Rules for source-set layout, expect/actual, cinterop (iOS/Native),
and JNI (JVM/Android). Verified 2026-07-22 against the cited pages
unless marked blog-grade.

## expect/actual discipline

The official expect/actual reference
(https://kotlinlang.org/docs/multiplatform/multiplatform-expect-actual.html):

> "As a general rule, rely on standard language constructs wherever
> possible instead of using expected and actual declarations"

> "Expected and actual classes are in Beta. They are almost stable,
> but migration steps may be required in the future."

Interfaces are the preferred alternative because they allow multiple
implementations per platform and easy fake substitution in tests
(same page). expect/actual *functions, properties, and typealiases*
are stable; only expect/actual **classes** carry the Beta caveat and
the default compiler warning (suppressed via
`-Xexpect-actual-classes`, which this repo sets).

**Audit rules:**

- Any expect/actual **class** is a checkable smell requiring recorded
  justification. This repo's justification: the driver mirrors
  `androidx.sqlite`'s own expect-class contract, and the platform
  interfaces (`SQLiteDriver` etc.) already exist upstream — an
  interface indirection of our own would duplicate them
  (`androidx-sqlite` skill; ARCHITECTURE.md D1). The stability risk
  stands regardless: a Kotlin upgrade may require migration steps.
- Every expect declaration needs a matching actual in the same
  package for every platform source set; actuals may be hoisted into
  intermediate source sets (e.g. one actual in a shared iOS set)
  rather than duplicated per leaf target (same page).
- `actual typealias` to a platform type is a legitimate pattern for
  bridging platform APIs without wrapper duplication (same page).

## Source-set layout and file naming

The coding conventions recommend platform suffixes on files holding
top-level declarations in platform source sets — and none in common
(https://kotlinlang.org/docs/coding-conventions.html):
`commonMain/kotlin/Platform.kt`, `jvmMain/kotlin/Platform.jvm.kt`,
`androidMain/kotlin/Platform.android.kt`. Rationale: avoids JVM
file-facade class collisions. Scope caveat: applies to files with
top-level declarations, not every file; single-source rule, verified
2-1 — treat as convention, not mandate. This repo conforms
(`DoltLiteDriver.jvmAndroid.kt`, `DoltLiteDriver.native.kt`,
`loadNativeLibrary.android.kt`).

Community-consolidated (blog-grade, derived from official KMP docs:
https://github.com/mmiani/kotlin-kmp-claude-agent-skills): shared
code lives as high in the source-set hierarchy as it validly can;
platform-specific code confined to the edges; no platform code in
`commonMain`. Use as a review heuristic.

## cinterop safety (iosMain / nativeMain)

From https://kotlinlang.org/docs/native-c-interop.html (page dated
2025-12-08, all verified verbatim):

> "All Kotlin declarations generated by the cinterop tool from C
> libraries should have the `@ExperimentalForeignApi` annotation"

- **Greppable surface map:** every foreign-API use site opts in via
  `@OptIn(ExperimentalForeignApi::class)` — grep for it to enumerate
  the interop surface. False-negative mode: a module-wide `-opt-in`
  Gradle flag hides per-site markers; check `build.gradle.kts` first.
  (This repo opts in per-file, e.g. `DoltLiteDriver.native.kt`.)
- **Pointer casts:**

  > "As is with C, these .reinterpret casts are unsafe and can
  > potentially lead to subtle memory problems"

  Every `.reinterpret<T>()` occurrence is an audit target: verify the
  target type against the C declaration.
- **Allocation regimes:** `memScoped` memory "will be automatically
  freed after leaving the scope" — a pointer escaping the block is a
  use-after-free defect. `nativeHeap.alloc` "requires memory to be
  freed manually" via a matching `nativeHeap.free` — an unmatched
  allocation is a leak. Audit each allocation site for which regime
  it uses and whether its lifetime obligation is met.

## Kotlin/Native memory model

https://kotlinlang.org/docs/native-memory-manager.html:

> "Objects are stored in a shared heap and can be accessed from any
> thread."

The legacy model is gone: freezing was deprecated in 1.7.20, the old
memory manager removed in 1.9.20
(https://kotlinlang.org/docs/whatsnew1920.html), and `freeze()` does
not exist in Kotlin 2.x
(https://kotlinlang.org/docs/native-migration-guide.html).

**Audit rules:** do NOT require `freeze()`/`@SharedImmutable`/
thread-confinement patterns in current native code — flag any that
remain as dead legacy. Ordinary shared-mutable-state thread-safety
review still applies (see `coroutines-audit.md`). Note this repo's
own confinement contract is stricter than the memory model:
connections are not thread-safe (`SQLITE_THREADSAFE=2`), confinement
is Room's job (ARCHITECTURE.md D1; `sqlite-c-api` skill).

## JVM JNI hygiene (jvmAndroidMain)

No authoritative JetBrains/Google JNI-for-Kotlin hygiene doc
surfaced in the research pass (open question) — the citable rules
here are blog-grade (https://jonnyzzz.com/blog/2019/12/15/jni-kotlin/)
plus this repo's established patterns:

- Native symbols follow `Java_<package>_<class>_<method>` mangling
  OR are registered explicitly. This repo uses `RegisterNatives`
  from `JNI_OnLoad` (`DoltLiteNative.kt`), which turns signature
  mismatches into load-time failures instead of first-call crashes —
  preserve that property when adding methods: every new `external
  fun` needs a matching entry in the `RegisterNatives` table in
  `doltrooms_jni.cpp`.
- JNI-exported C functions keep the leading `JNIEnv*` and
  `jclass`/`jobject` parameters even when unused — the calling
  convention requires them.
- The C-side review (local-reference management, exception checks
  after JNI calls, out-parameter conventions) belongs to the
  `sqlite-c-api` and `androidx-sqlite` skills' call maps; this repo's
  binding style (result-code returns, `rcOut`/`stmtOut` out-arrays,
  handles as `Long`) is documented in `DoltLiteNative.kt` KDoc.
- Cross-target correctness of bindings is established by one shared
  test suite run on every target (blog-grade pattern:
  https://medium.com/kodein-koders/native-dependency-in-kotlin-multiplatform-part-1-architecture-7fb9ded56bf1)
  — this repo's `commonTest` abstract conformance suites subclassed
  per target are exactly this; keep new driver behavior covered
  there, not in per-platform one-offs (ARCHITECTURE.md D7).
